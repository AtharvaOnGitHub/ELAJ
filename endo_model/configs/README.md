# endo_model/configs

Config dataclasses and the YAML loader.  No hyperparameter is hardcoded in any model class — everything comes through here.

## Files

| File | Responsibility |
|---|---|
| `model_config.py` | `ModelConfig` — all architectural dimensions; sub-configs for each component (`GroupEncoderConfig`, `AggregationEncoderConfig`, `DecoderConfig`, `CCEConfig`, `DABConfig`) |
| `training_config.py` | `TrainingConfig` — optimiser, scheduler, KL annealing, checkpoint resume |
| `data_config.py` | `DataConfig` — vocabulary path, studies CSV path, split strategy |
| `experiment_config.py` | `ExperimentConfig` — root config that references the three above plus dot-path overrides |
| `__init__.py` | `load_experiment()` and `apply_override()` — loader and override logic |

## Config ownership principle

**Architectural component configs belong to `ModelConfig`.**  CCE, DAB, and DSBN parameters describe how model components behave — they belong with the architecture, not the training loop.  The experiment YAML uses overrides (e.g. `model.cce.enabled: true`) as a lightweight on/off switch; the component-level detail stays in the model YAML.

**Training loop mechanics belong to `TrainingConfig`.**  Optimiser, scheduler, gradient clipping, early stopping, and KL annealing are training procedures, not model properties.

## Data flow

```
configs/experiment/full_model.yaml
    │
    └── load_experiment(path, root)
            │  reads experiment YAML
            ├── loads configs/model/ELAJ_small.yaml      → ModelConfig
            ├── loads configs/training/base.yaml         → TrainingConfig
            ├── loads configs/data/corpus.yaml           → DataConfig
            └── applies dot-path overrides
```

`load_experiment` is the single entry point. Scripts call it at startup and pass the resulting dataclasses to `Trainer` and model constructors.

## Class responsibilities

### `ModelConfig`
Holds all architectural dimensions: `d_token`, `d_agg`, `d_gauss`, `n_vMF`, `d_dec`.  Two derived read-only properties:
- `d_z = d_gauss + 2 * n_vMF` — total latent dimension. The `2 *` is because each vMF dim is stored as `(cos θ, sin θ)` — two Cartesian coordinates. See [`model_config.py`](model_config.py) for the full explanation.
- `d_gene_rep = d_token + d_agg` — gene representation fed to the decoder.

`n_groups` and `n_studies` are `None` until the data pipeline sets them at runtime.

### `CCEConfig` (owned by `ModelConfig`)
Controls the Contrastive Cell Embeddings objective: `enabled` (on/off), `temperature`, `lambda_cce`.  The Trainer skips the second forward pass entirely when `enabled=False`.

### `DABConfig` (owned by `ModelConfig`)
Controls the Domain Adversarial Batch correction classifier: `lambda_dab`, `hidden_dim`.

### `TrainingConfig`
Controls the training loop. `KLAnnealingConfig` controls the beta ramp.  `resume_from_checkpoint` accepts a path to a `.pt` checkpoint; `null` means train from scratch.

### `DataConfig`
`studies_csv` points to a CSV generated by the preprocessing pipeline (see `docs/corpus_csv_spec.md`).  `resolve_studies(root)` loads the CSV lazily — the file does not need to exist when the config is loaded.  Inline `studies` list is available as a fallback for quick testing without a real corpus.

### `ExperimentConfig`
The root config. Its `overrides` dict supports nested dot-path keys.

## Override format

```yaml
# configs/experiment/my_run.yaml
overrides:
  model.cce.enabled: true         # toggle CCE on
  model.cce.temperature: 0.05     # adjust CCE temperature
  training.max_epochs: 50         # shorter run
  training.resume_from_checkpoint: outputs/epoch_10.pt
```

Keys must start with `model.`, `training.`, or `data.`. Nested keys work to any depth.

## Default seed

The personal watermark seed is **24**.  It appears only in dataclass defaults (`SplitConfig.seed`, `ExperimentConfig.seed`) — not hardcoded anywhere in model or training logic.

## Design choices

- **No external dependency for config loading.** Every `from_dict` is handwritten with explicit field access and defaults.
- **`from_dict` always has defaults.** Partial YAML configs work; missing keys fall back to defaults.
- **CSV loading is deferred.** `DataConfig.from_dict` stores the CSV path without reading the file. `resolve_studies(root)` is called by the Dataset at data-loading time, so configs can be constructed without the corpus being available.
- **`load_experiment` resolves sub-config paths relative to the project root**, not relative to the experiment YAML.

## Unit tests

`tests/test_configs/test_configs.py` covers:
- All config defaults and derived properties (including `d_z = d_gauss + 2 * n_vMF` verified at multiple values)
- `from_dict` for all classes, partial and full inputs
- `CCEConfig` and `DABConfig` defaults and from_dict
- `TrainingConfig` does not contain CCE fields (assertion that ownership moved to ModelConfig)
- `KLAnnealingConfig` defaults and from_dict
- `resume_from_checkpoint` round-trip
- `SplitConfig` default seed = 24
- CSV loading: happy path, missing file, missing column, unsorted rows
- `resolve_studies` with both inline and CSV sources
- `apply_override` at shallow and nested depths
- Full YAML round-trips for all experiment configs
- CCE toggle via override (`model.cce.enabled: true`)
- Default seed watermark assertion
